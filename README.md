# Практические работы по фреймворкам

В репозитории лежат четыре небольших учебных проекта.  
Их цель — не сделать большое приложение, а отдельно разобрать несколько механизмов, которые часто встречаются во фреймворках: конвейер запросов, модули, DI-контейнер и плагины.

## Состав работ

| № | Проект | Технология | Что проверяется |
|---|--------|------------|-----------------|
| 1 | Мини веб-служба | C# / ASP.NET Core | Middleware, Swagger, обработка ошибок |

## Практическая работа №1  
### Мини веб-служба на ASP.NET Core

В первой работе сделана небольшая веб-служба с тремя основными запросами:

```http
GET  /api/items
GET  /api/items/{id}
POST /api/items
```

Сервис хранит данные в памяти процесса. База данных здесь не используется специально, чтобы не усложнять пример и сосредоточиться на конвейере обработки запроса.

Запрос проходит через несколько middleware:

```text
RequestId → обработка ошибок → логирование → замер времени → endpoint
```

Каждый middleware отвечает только за свою задачу. За счёт этого код endpoint'ов остаётся проще: они занимаются только логикой работы с элементами, а не логированием и обработкой исключений.

### Запуск

Открыть решение в Visual Studio 2022:

```text
Pr1.MinWebService.sln
```

Назначить проект `Pr1.MinWebService` запускаемым и нажать `F5`.

Swagger откроется по адресу:

```text
https://localhost:XXXXX/swagger
```

Порт нужно смотреть в консоли Visual Studio.

### Что проверить

1. `GET /api/items` — получение списка.
2. `GET /api/items/1` — получение элемента.
3. `GET /api/items/999` — ошибка с `requestId`.
4. `POST /api/items` — создание нового элемента.
5. Логи в консоли — там должны быть `requestId`, путь запроса, статус ответа и время выполнения.

### Что получилось

Получилась простая веб-служба, в которой видно, как запрос идёт через конвейер обработки. Ошибки возвращаются в одном формате, а в логах можно найти конкретный запрос по `requestId`.

### Что можно улучшить

- добавить базу данных;
- добавить авторизацию;
- добавить редактирование и удаление;
- писать логи не только в консоль, но и в файл;
- расширить Swagger-описания.

## Практическая работа №2  
### Модульное приложение на Node.js

Во второй работе приложение разбито на модули. У каждого модуля есть единый контракт:

```js
module.exports = {
  name: 'module-name',
  requires: [],
  register(container) {},
  init(container) {}
};
```

`name` — имя модуля.  
`requires` — список модулей, которые должны быть запущены раньше.  
`register` — регистрация сервисов в DI-контейнере.  
`init` — запуск модуля.

В проекте есть несколько модулей:

```text
data-module
validation-module
report-module
export-module
```

Например, модуль отчёта зависит от модуля данных, потому что отчёт строится на основе списка задач.

### Запуск

```bash
npm install
npm start
```

Swagger:

```text
http://localhost:3000/swagger
```

### Проверка

```bash
npm test
```

Отдельные сценарии:

```bash
npm run start:missing
npm run start:cycle
```

`start:missing` проверяет ситуацию, когда в конфигурации не хватает нужного модуля.  
`start:cycle` проверяет циклическую зависимость.

### Что проверить в Swagger

```http
GET  /api/modules
GET  /api/tasks
GET  /api/report
POST /api/tasks
```

После создания новой задачи отчёт должен измениться. Это показывает, что модули не просто загружаются, а реально влияют на поведение приложения.

### Что получилось

Приложение читает список модулей из конфигурации, загружает их из папки `modules`, проверяет зависимости и строит порядок запуска. Если конфигурация неправильная, приложение не падает случайной ошибкой, а пишет понятную причину.

### Что можно улучшить

- добавить версии модулей;
- проверять совместимость версий;
- добавить асинхронную инициализацию;
- сделать загрузку внешних модулей;
- добавить больше тестов на разные схемы зависимостей.

## Практическая работа №3  
### DI-контейнер с жизненными циклами

В третьей работе реализован свой DI-контейнер. Он поддерживает три жизненных цикла:

| Lifetime | Как работает |
|----------|--------------|
| singleton | один объект на всё приложение |
| transient | новый объект при каждом запросе к контейнеру |
| scoped | один объект на один HTTP-запрос |

Главная часть работы — показать, что состояние объекта зависит от выбранного жизненного цикла.

### Запуск

```bash
npm install
npm start
```

Swagger:

```text
http://localhost:3003/swagger
```

Эксперимент:

```http
GET /api/experiment
```

Демонстрация параллельных запросов:

```bash
npm run demo
```

Тесты:

```bash
npm test
```

### Что происходит в эксперименте

Внутри одного запроса контейнер несколько раз получает сервисы с разными lifetime.

Ожидаемое поведение:

```text
singleton  → один и тот же объект
transient  → каждый раз новый объект
scoped     → один объект внутри запроса, но другой в новом запросе
```

При параллельных запросах хорошо видно, почему `scoped` важен для серверной разработки. Он не даёт состоянию одного запроса попасть в другой.

### Что получилось

Контейнер умеет регистрировать зависимости, создавать объекты и учитывать их жизненный цикл. На примере запросов видно, что `singleton`, `transient` и `scoped` ведут себя по-разному.

### Что можно улучшить

- добавить освобождение ресурсов после завершения scope;
- добавить проверку циклических зависимостей между сервисами;
- добавить асинхронные фабрики;
- добавить логирование создания объектов;
- сделать небольшой нагрузочный тест.

## Практическая работа №4  
### Реестр плагинов

Четвёртая работа посвящена плагинной системе. Плагин — это отдельное расширение, которое можно подключить к приложению без изменения ядра.

Пример плагина:

```js
module.exports = {
  name: 'report-plugin',
  dependsOn: ['logging-plugin'],

  init(context) {
    context.log('report-plugin started');
  }
};
```

У плагина есть имя, список зависимостей и метод инициализации. Приложение собирает все плагины в реестр и запускает их в правильном порядке.

### Запуск

```bash
npm install
npm start
```

Swagger:

```text
http://localhost:3004/swagger
```

Проверка отсутствующего плагина:

```bash
npm run start:missing
```

Проверка цикла:

```bash
npm run start:cycle
```

Тесты:

```bash
npm test
```

### Что проверить в Swagger

```http
GET /api/plugins
GET /api/startup-order
GET /api/status
GET /api/logs
```

Особенно важен `GET /api/startup-order`. Он показывает, что плагины запускаются не в случайном порядке, а по зависимостям.

### Что получилось

Приложение умеет загружать плагины, проверять зависимости, строить порядок запуска и находить ошибки в конфигурации. Если плагина нет или зависимости образуют цикл, запуск останавливается с понятным сообщением.

### Что можно улучшить

- добавить версии плагинов;
- проверять совместимость плагинов;
- добавить отключение плагинов через конфиг;
- добавить откат, если один из плагинов не запустился;
- добавить интерфейс для просмотра подключённых плагинов.

## Общий вывод

В этих работах были разобраны четыре механизма, которые часто используются внутри фреймворков:

1. конвейер обработки HTTP-запроса;
2. загрузка модулей;
3. управление временем жизни зависимостей;
4. запуск плагинов с учётом зависимостей.

Главный результат — стало понятнее, что фреймворк занимается не только вызовом функций. Он управляет тем, как проходит запрос, когда создаются объекты, как подключаются расширения и в каком порядке они запускаются.

Такой подход делает приложение более понятным: прикладная логика остаётся отдельно, а техническая инфраструктура находится в одном месте.
