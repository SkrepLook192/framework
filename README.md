# Практическая работа №1
# C# / ASP.NET Core: мини веб-служба, Swagger и конвейер обработки запросов

## 1. Главная идея

Первая работа показывает, как HTTP-запрос проходит через веб-приложение.

Пользователь отправляет запрос, например:

```http
GET /api/items
```

Но запрос не попадает сразу в endpoint. Сначала он проходит через цепочку middleware:

```text
Запрос пользователя
    ↓
RequestIdMiddleware
    ↓
ExceptionHandlingMiddleware
    ↓
RequestLoggingMiddleware
    ↓
PerformanceMiddleware
    ↓
Endpoint /api/items
    ↓
Ответ пользователю
```

Каждый middleware делает одну понятную вещь:

```text
RequestIdMiddleware              задаёт идентификатор запроса
ExceptionHandlingMiddleware      ловит ошибки и делает единый JSON-ответ
RequestLoggingMiddleware         пишет входящий запрос и исходящий ответ в журнал
PerformanceMiddleware            измеряет время выполнения запроса
```

Главная мысль работы:

```text
Веб-служба должна быть устроена как понятный конвейер обработки запроса, а не как набор случайных функций.
```

## 3. Какие запросы есть

Обязательных запроса всего три:

```text
GET  /api/items
GET  /api/items/{id}
POST /api/items
```

### GET /api/items

Возвращает список учебных задач.

Пример ответа:

```json
[
  {
    "id": 1,
    "title": "Изучить middleware в ASP.NET Core",
    "subject": "C#",
    "estimatedHours": 2,
    "isCompleted": false,
    "createdAt": "2026-05-01T10:00:00Z"
  }
]
```

### GET /api/items/{id}

Возвращает одну задачу по id.

Пример:

```http
GET /api/items/1
```

Если задачи нет:

```http
GET /api/items/999
```

возвращается ошибка:

```json
{
  "errorCode": "ITEM_NOT_FOUND",
  "message": "Элемент с идентификатором 999 не найден.",
  "requestId": "..."
}
```

### POST /api/items

Создаёт новую задачу.

Тело запроса:

```json
{
  "title": "Подготовить практическую работу",
  "subject": "Фреймворки",
  "estimatedHours": 3
}
```

Если данные корректные, сервер создаёт задачу и возвращает её с новым `id`.

## 4. Что происходит в коде

### StudyTask

`StudyTask` — главный объект предметной области. Он описывает учебную задачу.

Поля:

```text
Id              уникальный номер задачи
Title           название задачи
Subject         предмет
EstimatedHours  примерное количество часов
IsCompleted     выполнена ли задача
CreatedAt       дата создания
```

### CreateStudyTaskRequest

Это модель входных данных для `POST /api/items`.

Клиент передаёт только:

```text
Title
Subject
EstimatedHours
```

А сервер сам создаёт:

```text
Id
CreatedAt
IsCompleted
```

Это правильно, потому что клиент не должен сам назначать id или дату создания.

### ErrorResponse

Это единый формат ошибки:

```json
{
  "errorCode": "...",
  "message": "...",
  "requestId": "..."
}
```

Он нужен, чтобы все ошибки в API выглядели одинаково.

### Репозиторий

Репозиторий отвечает за хранение данных.

В этой работе база данных не используется. Данные хранятся в памяти процесса.

Это значит:

```text
приложение запущено       данные есть
приложение перезапущено   данные сбросились
```

Для учебной работы это нормально.

### ConcurrentDictionary

Для хранения используется потокобезопасная коллекция.

Это нужно, потому что веб-сервер может обрабатывать несколько запросов одновременно.

Если два пользователя одновременно создают задачи, обычный `List` может быть опасен. `ConcurrentDictionary` лучше подходит для параллельного доступа.

## 5. Middleware подробно

### RequestIdMiddleware

Создаёт уникальный идентификатор запроса.

Он нужен, чтобы связать:

```text
запрос пользователя
логи
ошибку
ответ сервера
```

Если пользователь получил ошибку, по `requestId` можно найти её в журнале.

### ExceptionHandlingMiddleware

Ловит необработанные ошибки и превращает их в аккуратный JSON.

Пользователь не должен видеть внутренний stack trace приложения.

### RequestLoggingMiddleware

Пишет в консоль входящий запрос и исходящий ответ:

```text
Входящий запрос. requestId=abc method=GET path=/api/items
Исходящий ответ. requestId=abc status=200
```

### PerformanceMiddleware

Измеряет время выполнения запроса:

```text
Время выполнения запроса. requestId=abc elapsedMs=15
```

# Практическая работа №2
# JavaScript / Node.js: модульное приложение и DI

## 1. Главная идея

Вторая работа показывает, как сделать приложение, которое можно расширять модулями.

Есть ядро приложения. Оно не должно знать детали каждого модуля.

Ядро знает только общий контракт:

```js
module.exports = {
  name: 'module-name',
  requires: ['other-module'],
  register(container) {},
  init(container) {}
};
```

То есть каждый модуль имеет:

```text
name        имя модуля
requires    список зависимостей
register    регистрация сервисов в DI-контейнере
init        инициализация модуля
```

Главная мысль:

```text
Новое поведение добавляется через новый модуль, а не через переписывание ядра.
```

## 5. Как работает порядок запуска

Если модуль зависит от другого модуля, зависимость должна быть запущена раньше.

Например:


```text

```

Даже если в конфиге они записаны иначе, ядро строит правильный порядок через топологическую сортировку.

```js
container.registerSingleton('dataService', () => {
```

Главная идея:

```text
Модули не создают зависимости вручную. Они получают их из контейнера.
```

# Практическая работа №3
# JavaScript / Node.js: DI-контейнер с singleton, transient, scoped

## 1. Главная идея

Третья работа показывает, как жизненный цикл зависимости влияет на состояние приложения.

Есть три жизненных цикла:

```text
singleton  один объект на всё приложение
transient  новый объект при каждом обращении
scoped     один объект на один HTTP-запрос
```

Главная мысль:

```text
Scoped важен для серверной разработки, потому что изолирует состояние одного запроса от другого.
```

---

## 2. Как запустить работу №3

```bash
cd Task3_JS_DI_Lifetimes

npm start
```

Открыть:

```text
http://localhost:3003/api/experiment
```

Демонстрация параллельности:

```bash
npm run demo
```

Тесты:

```bash
npm test
```

---

## 3. Что такое DI-контейнер

DI-контейнер создаёт объекты за приложение.

Вместо:

```js
const service = new SomeService();
```

мы пишем:

```js
container.register('serviceName', factory, lifetime);
```

А потом:

```js
const service = container.resolve('serviceName');
```

Контейнер сам решает, что вернуть.

---

## 4. Как хранятся зависимости

Внутри контейнера есть реестр:

```js
registrations = {
  singletonCounter: {
    factory: () => new Counter(),
    lifetime: 'singleton'
  },
  transientCounter: {
    factory: () => new Counter(),
    lifetime: 'transient'
  },
  scopedCounter: {
    factory: () => new Counter(),
    lifetime: 'scoped'
  }
}
```

Контейнер хранит:

```text
имя сервиса
как его создать
сколько он должен жить
```

---

## 5. Singleton

`singleton` создаётся один раз на всё приложение.

Плюсы:

```text
экономит ресурсы
удобен для общей конфигурации
подходит для логгера
```

Минусы:

```text
опасен для хранения данных пользователя
опасен для requestId
может привести к протеканию состояния между запросами
```

---

## 6. Transient

`transient` создаётся каждый раз заново.

Если вызвать два раза:

```js
scope.resolve('transientCounter')
scope.resolve('transientCounter')
```

будут два разных объекта.

Плюсы:

```text
нет общего состояния
безопасен для временных объектов
```

Минусы:

```text
может создавать слишком много объектов
не подходит, если нужно общее состояние внутри запроса
```

---

## 7. Scoped

`scoped` создаётся один раз на один scope.

В веб-приложении один scope — это один HTTP-запрос.

```text
Запрос A:
  scopedService #1
  scopedService #1

Запрос B:
  scopedService #2
  scopedService #2
```

Внутри одного запроса объект один и тот же. Между запросами разные объекты.

Это подходит для:

```text
requestId
текущего пользователя
контекста операции
транзакции
накопления данных одного запроса
```

---

## 8. Endpoint /api/experiment

Endpoint показывает разницу между lifetime.

Он получает сервисы несколько раз:

```js
const singletonA = scope.resolve('singletonCounter');
const singletonB = scope.resolve('singletonCounter');

const transientA = scope.resolve('transientCounter');
const transientB = scope.resolve('transientCounter');

const scopedA = scope.resolve('scopedCounter');
const scopedB = scope.resolve('scopedCounter');
```

Ожидаемый результат:

```text
singletonA === singletonB   true
transientA === transientB   false
scopedA === scopedB         true
```

Но между разными запросами:

```text
singleton общий
scoped разный
transient всегда новый
```

Команда:

Цель — показать:


```text
1. core-plugin
2. logging-plugin
3. report-plugin
4. export-plugin
```

Даже если в конфиге они указаны иначе, ядро должно построить правильный порядок.

---

## 6. Почему порядок важен

Если `report-plugin` использует логгер, а `logging-plugin` ещё не запущен, то `report-plugin` может сломаться.

Поэтому сначала запускаются зависимости, потом сам плагин.

---

## 7. Топологическая сортировка

Простыми словами:

```text
1. Берём плагин.
2. Смотрим его зависимости.
3. Сначала запускаем зависимости.
4. Потом запускаем сам плагин.
5. Если в текущей цепочке снова встретили тот же плагин — это цикл.
```

---

## 8. Ошибка отсутствующего плагина

Если:

```text
report-plugin зависит от logging-plugin
```

но `logging-plugin` не найден, приложение пишет понятную ошибку.

Это лучше, чем непонятное падение программы.

---

## 9. Ошибка циклической зависимости

Цикл:

```text
plugin-a зависит от plugin-b
plugin-b зависит от plugin-c
plugin-c зависит от plugin-a
```

Такое невозможно запустить.

Приложение должно остановиться и написать:

```text
Обнаружен цикл зависимостей
```
